Quiero que trabajes sobre mi proyecto actual para convertirlo en una base reutilizable que permita clonar la web “recambios-aspirador” hacia otros nichos, manteniendo al principio el acceso a esos nuevos contenidos desde la propia web principal actual.

OBJETIVO REAL
- Ya tengo una web funcional orientada a recambios de aspiradores/robots aspiradores.
- Quiero reutilizar la misma arquitectura para lanzar otros nichos compatibles con el mismo patrón:
  - marca
  - modelo
  - categorías de recambio/accesorio
  - problemas
  - enlaces a AliExpress
- En una primera fase NO quiero dominios nuevos.
- Quiero que esos nuevos nichos cuelguen inicialmente de la web actual y se acceda a ellos desde recambios-aspirador.
- Más adelante, si un nicho demuestra tráfico suficiente, ya se podrá independizar.

CONTEXTO TÉCNICO REAL
El pipeline actual que sí usamos es este:
1. editar:
   - data/catalog_parts.yaml
   - data/catalog_brands.yaml
   - data/catalog_skus.yaml
2. ejecutar:
   - python tools/build_catalog.py
   - python tools/generar.py --force
   - python tools/sync_ofertas.py --no-cache --force

Notas importantes:
- data/aspiradores.yaml es generado
- data/ofertas.yaml es generado
- no quiero editar outputs generados a mano
- armageddon_catalog.py queda fuera de este trabajo
- no quiero unificar scripts todavía
- el sistema actual funciona y no quiero romperlo

OBJETIVO DE ARQUITECTURA
Quiero que el proyecto deje de estar pensado solo para “aspiradores” y pase a ser una base multi-nicho, pero sin rehacerlo desde cero.

La idea es:
- seguir teniendo una web principal
- dentro de esa web principal, poder colgar nuevos nichos experimentales
- cada nicho debe poder reutilizar casi toda la lógica actual
- el acceso inicial a esos nichos debe salir desde recambios-aspirador
- después ya decidiré si los independizo o no

LO QUE QUIERO QUE IMPLEMENTES

1. Introducir el concepto de NICHO o VERTICAL en la arquitectura
Quiero que exista una forma clara de definir nichos/verticales en los datos fuente.

Ejemplos de verticales:
- aspiradores
- afeitadoras
- herramientas-bateria
- cafeteras

Cada vertical debe poder tener:
- key
- label
- slug
- categorías permitidas
- packs base opcionales
- quizá copy o textos de entrada opcionales

2. Asociar marcas y modelos a un vertical
Cada marca/modelo debe pertenecer a un vertical.
No quiero que todo siga asumiendo implícitamente “aspiradores”.

3. Mantener el vertical actual funcionando
Todo lo que ya existe debe seguir funcionando.
No quiero romper:
- build_catalog.py
- generar.py
- sync_ofertas.py
- layouts actuales
- contenido ya generado

4. Permitir que el contenido de otros nichos cuelgue inicialmente de la web actual
No quiero todavía dominios separados.
Quiero que los nuevos nichos se publiquen como nuevas secciones o ramas dentro de la web actual.

Ejemplo conceptual:
- /aspiradores/
- /afeitadoras/
- /herramientas-bateria/

o la estructura que veas más limpia y segura para el repo real.

5. Preparar acceso desde la web principal actual
Quiero que desde la web actual se pueda enlazar a estos nuevos nichos experimentales.
No busco aún una mega home multi-nicho compleja, pero sí una forma clara de acceder a ellos desde el proyecto actual.

6. Reutilizar al máximo la lógica ya existente
Quiero que se reutilice:
- la estructura YAML
- la generación de catálogo
- la generación de stubs
- la lógica de ofertas
- la lógica de recambios/categorías/problemas

No quiero duplicar el sistema entero por cada nicho.

QUÉ NO QUIERO
- no quiero reescribir el proyecto desde cero
- no quiero cambiar de stack
- no quiero que me propongas una arquitectura abstracta excesiva
- no quiero tocar armageddon_catalog.py
